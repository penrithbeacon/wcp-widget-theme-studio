WCP Theme Studio Guide

The definitive reference for creating, editing, and distributing themes across all WCP widgets and applications.

The WCP Theme Studio is the official theming tool for the Widget Context Protocol ecosystem. It ships with 15 built-in themes — including the 3 Penrith Beacon native themes — and a full editor for creating custom themes from scratch.

Every WCP widget receives its visual identity through 81 CSS custom properties injected into the iframe's :root element by the host dashboard. These tokens control everything from colours and typography to spacing, shadows, and motion. The Theme Studio lets you edit all of these visually, preview them in real time, and export them for use across any orchestration.

How themes work When a WCP host renders a widget, it injects a <style> block setting all 81 --wcp-* variables on :root. The widget's CSS references these variables (with fallbacks), so it automatically adapts to any theme without code changes.

Quick Start: Creating Your First Theme

The Three Panels

The Theme Studio Full editor is organised into three panels:

  • Left panel — Theme list. Browse the 15 built-in themes and any custom themes you've created. Click a theme to preview it.
  • Centre panel — Live preview. Shows a Demo tab (sample UI elements rendered with the current theme) and an About tab (theme metadata). Switch between them using the tab buttons.
  • Right panel — Theme editor. When you select a theme and click Edit (or create a new one), this panel shows all editable fields organised by category.

Step 1: Browse Themes

Click any theme in the left panel to see it rendered in the preview. The colour swatches next to each name give a quick visual summary of that theme's palette.

Step 2: Create a New Theme

  1. Click "+ New Custom Theme" at the bottom of the left panel.
  2. A new untitled theme appears with default values (cloned from the current Penrith Beacon WCP Dark theme).
  3. The right panel immediately opens in edit mode — start changing colours.

Step 3: Edit Colours & Properties

The editor organises all 81 tokens into sections. Each field shows a colour picker (for colour tokens) or a text input (for size, font, shadow tokens). Changes preview instantly in the centre panel.

Step 4: Save Your Theme

Click Save in the editor footer. Your custom theme is now stored locally in the container and appears in the theme list with a pencil icon. You can rename it by editing the Name field at the top of the editor.

Step 5: Apply to Your Dashboard

Click the "Add to Dashboard" button next to any theme in the list. This sends the theme to the host dashboard via WCP postMessage, applying it immediately to all widgets in the current orchestration.

Token Design Rationale

The 81 tokens aren't random — they follow deliberate patterns that make choosing the right token intuitive once you understand the underlying model.

The Surface Stack

Think of your UI as layers stacked on a desk. Each layer is visually distinct from the one below it:

┌──────────────────────────────── bg (deepest) ────────────────────┐ │ │ │ ┌────────────────────── surface (card) ────────────────────┐ │ │ │ │ │ │ │ ┌──────────── surface-raised (input) ────────────┐ │ │ │ │ │ │ │ │ │ │ │ text content lives here │ │ │ │ │ │ │ │ │ │ │ └──────────────────────────────────────────────────┘ │ │ │ │ │ │ │ └───────────────────────────────────────────────────────────┘ │ │ │ └────────────────────────────────────────────────────────────────────┘
  • bg — The page itself. The deepest, darkest (or lightest) layer.
  • surface — A card or panel sitting on the page. One step up in visual elevation.
  • surface-raised — An input field, nested panel, or interactive element inside a card. Another step up.
  • surface-sunken — A recessed well (code block background, inset area). One step below surface.
  • overlay — A semi-transparent backdrop covering everything beneath a modal or dialog.
Rule of thumb If you're placing element A on top of element B, use the next surface level up. Page → surface. Card → surface-raised. Never skip levels.

The Text Hierarchy

Five text tokens cover every legibility need:

TokenWhen to useVisual weight
textHeadings, body copy, labels — anything the user must readFull contrast
text-mutedTimestamps, secondary info, descriptions — readable but subdued~60% contrast
text-disabledGreyed-out controls — signals "not interactive"~40% contrast
text-inverseText on a solid coloured background (usually same hue as bg)Full contrast (reversed)
linkClickable text — must be distinguishable from textBlue/accent

The "-on" Pattern

Whenever a token defines a coloured surface (like a button background), there's a matching -on token for text that sits on that surface. The theme author sets both to guarantee legibility regardless of the chosen palette.

Background tokenText-on tokenTypical use
--wcp-color-primary--wcp-color-primary-onBrand action buttons
--wcp-color-success--wcp-color-success-onLaunch / confirm / save buttons
Don't forget the -on tokens! If you set --wcp-color-success to a light colour (e.g. #90EE90), white text becomes illegible. Set --wcp-color-success-on to a dark colour like #1a3a1a to maintain contrast.

Status Colour Variants

Each status colour (success, warning, danger, info) can have up to three variants:

VariantPurposeExample use
--wcp-color-successThe colour itselfButton background, icon colour, inline indicator
--wcp-color-success-onText legible on that colourButton label text
--wcp-color-success-surfaceSubtle tinted backgroundStatus badge, notification banner background

This three-variant pattern gives widget developers everything they need for rich status UIs without hard-coding colours.

Import & Export

Exporting a Theme

There are several ways to share a theme:

  • Copy URL — Click the "Copy URL" button next to any theme. This copies a URL like http://localhost:3740/widget/api/theme/dracula that returns the theme as JSON. Any WCP dashboard can import from this URL.
  • Add to Dashboard — Sends the theme directly to the host dashboard via WCP postMessage. The theme is applied immediately.
  • Export as .wcpt — For distribution, use the WCP export format. The dashboard's export system can package themes as .wcpt files (WCP Theme archives).

Importing a Theme

  • Import .wcpt — Click "Import .wcpt" at the bottom of the left panel. Select a .wcpt file — the theme is added to your custom themes list.
  • From URL — In the dashboard settings, paste a theme URL. The dashboard fetches the JSON and applies it.
  • From another Theme Studio — If you run Theme Studio containers on multiple machines, each serves its themes as URLs. Copy a URL from one and import it on another.

Theme JSON Format

A theme is a JSON object with the following structure:

{
  "id": "my-custom-theme",
  "uuid": "unique-uuid-here",
  "name": "My Custom Theme",
  "builtin": false,
  "vars": {
    "--wcp-color-bg": "#0d1117",
    "--wcp-color-surface": "#161b22",
    "--wcp-color-surface-raised": "#1c2128",
    "--wcp-color-border": "#30363d",
    "--wcp-color-text": "#e6edf3",
    "--wcp-color-text-muted": "#8b949e",
    "--wcp-color-primary": "#f0883e",
    "--wcp-color-success": "#3fb950",
    "--wcp-color-danger": "#f85149",
    "--wcp-color-warning": "#d29922",
    "--wcp-color-info": "#58a6ff"
  }
}

The vars object contains only the tokens the theme author explicitly sets. The dashboard's buildWcpTokens() function derives the remaining tokens (e.g. -dim, -surface, -on variants) from these seed values automatically.

Tips & Best Practices

For Theme Authors

  • Always set both -on tokens. If you change primary or success to something light, update primary-on / success-on to a dark colour. The preview panel's Demo tab shows buttons — check them.
  • Test with the Demo tab. It renders sample buttons, cards, inputs, badges, and text in your theme. If anything looks wrong, adjust the corresponding token.
  • Keep surface layers distinct. Ensure there's visible contrast between bg, surface, and surface-raised. If they're too similar, cards blend into the page.
  • Don't skip surface levels. Put cards on surface, inputs on surface-raised. Jumping from bg straight to surface-raised breaks the visual hierarchy.
  • Use text-muted generously. Not everything needs full-contrast text. Timestamps, descriptions, and secondary labels should use text-muted to create visual hierarchy.

For Widget Developers

  • Always use fallback values. Write var(--wcp-color-bg, #0d1117) not just var(--wcp-color-bg). If a token isn't injected, your widget still renders correctly.
  • Never hard-code colours. Every colour in your CSS should reference a --wcp-* token. Hard-coded colours break when themes change.
  • Use the surface stack for depth. Your widget's outermost container uses bg or widget-bg. Cards inside use surface. Inputs inside cards use surface-raised.
  • Respect widget-* tokens for framing. Use widget-padding, widget-gap, and widget-radius for your widget's outer frame. This ensures consistent spacing across all widgets in an orchestration.
  • Test with multiple themes. The 15 built-in themes cover a wide range (light, dark, high contrast, warm, cool). If your widget looks good in all 15, it'll work with any custom theme.

Frequently Asked Questions

How many tokens does a theme need to define?

A minimal theme only needs 11-13 colour tokens (the ones in the Surfaces, Border, Text, Brand, and Status sections). The dashboard's buildWcpTokens() function automatically derives the remaining tokens — -dim, -surface, -on variants, disabled states, etc. — from your seed values.

However, you can override any of the 81 tokens explicitly for full control.

What's the difference between primary and success?

primary is your brand colour — used for the main action button, active tab indicators, selected states. It represents "this is the primary action."

success is semantic — it means "positive outcome." Used for launch buttons, save confirmations, online indicators. It's typically green but doesn't have to be.

Can I use different fonts in my theme?

Yes — set --wcp-font-family to any font stack. However, the font must be available on the user's system or loaded by your widget. WCP doesn't provide a font-loading mechanism, so stick to system fonts or web-safe stacks for maximum compatibility.

Why do some built-in themes only set ~13 values in their vars?

The theme's vars object contains only the values the author explicitly chose. Everything else is calculated by the dashboard's token resolution engine. For example, if you set --wcp-color-success to #3fb950, the engine automatically generates success-surface as rgba(63,185,80,.1) and success-on as #ffffff (unless you override them).

How do I make a high-contrast theme?

Maximise the difference between bg and text. Use pure black (#000000) for bg and pure white (#ffffff) for text. Set borders to full white. Remove shadows (none) and reduce radius values. See the built-in "Penrith Beacon WCP High Contrast" theme as a reference.

Can themes change spacing and shadows?

Yes — themes can set any of the 81 tokens, including spacing (--wcp-space-*), shadows (--wcp-shadow-*), radius (--wcp-radius-*), and motion (--wcp-duration-*, --wcp-easing-*). A "compact" theme might halve all spacing values. A "flat" theme might set all shadows to none.

What happens if a widget doesn't use a token?

Nothing — unused tokens are simply ignored. The host still injects all 81 tokens, but a simple widget might only reference 5-10 of them. There's no performance cost for unused tokens.

How do I test my theme across multiple widgets?

Create an orchestration with several different widgets on the same stave. Apply your theme and check each widget. The Demo tab in the Theme Studio preview also covers the most common UI patterns, so if it looks good there, it'll likely look good everywhere.
